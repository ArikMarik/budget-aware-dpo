# Knowledge Distillation vs. DPO — Clarification

This document explains that **knowledge distillation is not implemented** in this project and clarifies the difference between the two approaches.

---

## Summary

**Knowledge distillation is not implemented in this project.** The training uses **Direct Preference Optimization (DPO)** with a reference model, which is conceptually different from distillation.

---

## What This Project Uses: DPO (Direct Preference Optimization)

The training pipeline uses **DPO**, not knowledge distillation. The main components are:

1. **Policy model (π_θ)** — The model being trained (Qwen-2.5-0.5B + LoRA).
2. **Reference model (π_ref)** — A frozen copy of the same base model before training.
3. **Preference pairs** — (chosen, rejected) solution pairs from the dataset.

The loss uses log-ratios with the reference model:

```
log_ratio_chosen = policy_chosen_logps - reference_chosen_logps
log_ratio_rejected = policy_rejected_logps - reference_rejected_logps
```

The reference model is used to define the implicit reward in DPO, not to teach the policy via soft labels.

---

## What "Teacher" Means in This Codebase

The term **teacher** appears only in data metadata:

- **`teacher_token_count`** — A metadata field indicating the number of tokens in a solution (used for complexity classification and preference labeling).
- In OpenMathInstruct-2, solutions were generated by a large model (e.g., Llama3.1-405B), but that model is **not used during training**. Only the text solutions are used as preference data.

So "teacher" here refers to a data-format field, not a distillation teacher.

---

## Knowledge Distillation vs. DPO — Comparison

| Aspect | Knowledge Distillation | This Project (DPO) |
|--------|------------------------|---------------------|
| **Teacher** | Large model whose outputs are mimicked | No teacher model |
| **Student** | Small model trained to match teacher logits | Single policy model |
| **Objective** | Match teacher's output distribution | Prefer chosen over rejected in preference pairs |
| **Reference model** | Teacher | Frozen copy of initial policy (same size) |
| **Data** | Inputs + teacher outputs | (problem, chosen, rejected) pairs |

---

## Knowledge Distillation (Typical Setup)

- **Teacher:** Large model (e.g., 70B parameters).
- **Student:** Small model (e.g., 0.5B parameters).
- **Goal:** Student learns to match teacher's softmax probabilities over the vocabulary.
- **Loss:** KL divergence between teacher and student logits (or similar).

---

## DPO (This Project)

- **Policy:** Model being trained (same size as reference).
- **Reference:** Frozen copy of the same base model before training.
- **Goal:** Policy learns to prefer chosen over rejected in preference pairs.
- **Loss:** -log σ(reward_diff), where reward uses log-ratios vs. reference.

---

## Conclusion

There is no knowledge distillation. The project trains with **DPO** (and a budget-aware variant) using preference pairs and a frozen reference model. The reference model is not a teacher; it is used to compute log-ratios for the DPO loss.
